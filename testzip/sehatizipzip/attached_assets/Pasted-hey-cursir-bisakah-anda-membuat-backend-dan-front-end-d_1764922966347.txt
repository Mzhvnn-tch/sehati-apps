⭐ I. KOMPONEN SEHATI (SECARA TEKNIS DAN FUNGSIONAL)

SEHATI dibangun dari 4 lapisan besar:

---
🧱 1. Identity Layer (Self-Sovereign Health Identity)

Ini adalah “dompet identitas kesehatan” milik pasien.

Fungsinya:

Menyimpan private key (akses data kesehatan).

Menandatangani izin akses untuk dokter/RS.

Menghasilkan QR untuk berbagi data sementara.

Menyimpan pointer on-chain untuk semua rekam medis.

Teknologi:

Wallet crypto (MetaMask / Web3Auth / Privy).

DID (Decentralized Identifier).

Private Key AES/RSA untuk enkripsi data.

Kenapa penting:
➡️ Semua data kesehatan milik pasien, bukan RS.

---
🔒 2. Encryption Layer (Privacy Engine)

Semua data medis harus dienkripsi sebelum ditulis ke storage.

Teknologi:

AES-256-GCM untuk encrypt file/data.

RSA-2048 atau ECC untuk berbagi kunci ke dokter.

Key rotation (buat izin akses sementara).

QR code yang membawa encrypted access token.

Yang dienkripsi:

Riwayat obat

Hasil lab

Diagnosa dokter

Foto X-ray

Data sensitif (mental health, HIV, operasi, alergi)

Yang TIDAK dienkripsi:

Hash pointer (simpen di blockchain)

Timestamp

Signature dokter

RS mana yang telah mengeluarkan data

Ini aman → tidak ada data medis terang-benderang di blockchain.

---
⛓ 3. Blockchain Layer (Audit + Verifikasi)

Blockchain digunakan untuk verifikasi, bukan menyimpan data mentah.

Yang disimpan:

Hash data (verifikasi keaslian)

Alamat wallet pasien

Alamat wallet dokter/RS

Timestamp

Event pencatatan / pembaruan

Jejak izin akses (grant / revoke)

Smart Contract menyediakan:

registerRecord(hash, ipfsHash)

grantAccess(doctor, expiresAt)

revokeAccess(doctor)

verifyIssuer(doctorSignature)

Kenapa penting:
➡️ RS tidak bisa memalsukan rekam medis.
➡️ Pasien bisa audit siapa mengakses datanya.
➡️ Transparansi tanpa membocorkan isi.

---
🗄 4. Storage Layer (Encrypted Data Storage)

Data medis disimpan terenkripsi penuh di:

IPFS → decentralized, bagus untuk file medium

Arweave → permanen (cocok untuk rekam medis jangka panjang)

Data mentah tidak pernah disimpan di blockchain.

Setiap data punya struktur JSON terenkripsi:

{
"encrypted_data": "8d92ab...==",
"type": "lab_result",
"issued_by": "RS Jakarta Pusat",
"patient_id": "0x123...abc",
"timestamp": 1735898942
}

---
🖥 5. App Layer (Front-End untuk RS & Pasien)

Pasien App (Mobile/Web):

Login (Face ID / email / wallet)

Tampilkan riwayat kesehatan

Generate QR code untuk berbagi data

Kelola izin akses (Grant/Revoke)

Tampilkan log audit

Dokter App (Web/Tablet in RS):

Scan QR → otomatis decrypt data

Input hasil pemeriksaan baru

Submit ke IPFS → hash ke blockchain

---
🌐 6. Access Sharing Layer (Temporary Data Sharing)

Fitur paling penting untuk user awam:

QR Sharing System:

Pasien scan QR → dokter dapat izin baca 1x atau 1 jam.

Token bersifat sementara (expirable key).

Data yang dibagikan bisa parsial:

Alergi

Obat terakhir

Diagnosa terakhir

Catatan tertentu saja

Keunggulan: ➡️ Tidak semua data terbuka
➡️ Dokter hanya lihat yang relevan

---
🧬 7. Audit Trail Layer

Setiap aktivitas dicatat:

Siapa mengakses data

Jam berapa

RS mana

Hash record apa yang dibuka

Apakah akses berhasil atau ditolak

Transparansi penuh.
Privasi penuh.

---
🚀 II. ALUR CERITA — SKENARIO NYATA (SUPER DETAIL)

Skema: Si A keracunan dan masuk UGD rumah sakit baru.

Aku tuliskan alur dari sudut pandang pasien, dokter, dan sistem.

---
⭐ 1. Si A Keracunan (Kondisi Awal)

Mual, muntah, diare, tekanan darah turun.

Masuk UGD RS X (bukan RS langganan).

Dokter UGD tanya pertanyaan standar:

“Obat terakhir apa?”

“Ada alergi?”

“Riwayat penyakit?”

Si A lupa semua.

Masalah:
➡️ Bisa salah diagnosa → salah obat → berbahaya.

---
⭐ 2. Pasien Buka App SEHATI

Di HP, Si A buka app:

Login Face ID → masuk 1 detik.

Klik “Share Data Darurat”.

App otomatis menyiapkan dataset berikut:

Obat yang diminum 7 hari terakhir

Alergi

Diagnosa lama terkait pencernaan

Hasil lab minggu lalu

App generate QR code (valid 60 menit).

---
⭐ 3. Dokter Scan QR

Tablet dokter scan QR →
Backend proses:

Memvalidasi token izin.

Mengambil data terenkripsi dari IPFS.

Mendekripsi menggunakan:

Public key dokter

Temporary decrypt key pasien

Menampilkan UI data medis.
Dokter melihat:

Si A alergi penisilin

3 hari lalu minum antibiotik X

Pernah punya infeksi lambung

Hasil lab 1 minggu lalu normal

---
⭐ 4. Dokter Diagnosa Cepat

Karena data lengkap:

Dokter langsung cek obat apa yang aman.

Menghindari penisilin → hindari interaksi obat berbahaya.

Dokter menyimpulkan keracunan makanan + efek samping obat.

Perawatan dilakukan dalam 5 menit, bukan tanya-tanya 30 menit.

Ini literally bisa menyelamatkan nyawa.

---
⭐ 5. Dokter Menambah Rekam Medis Baru

Setelah perawatan:

Dokter buka panel “Tambah Catatan”

Isi: diagnosa + resep + saran

Minggirkan data sensitif (yang tidak perlu dishare global)

Sistem meng-enkripsi data itu:

pasien private key

dokter public key

File terenkripsi upload → IPFS

Hash dicatat ke blockchain.

Sekarang Si A punya catatan medis terbaru.

---
⭐ 6. Semua Tercatat di Audit Trail

Blockchain mencatat:

RS: RS X
Doctor: 0xABCD
Patient: 0x1234
Action: RecordAdded
Hash: Qm92asd...
Time: 2025-12-03T01:30Z

Tidak ada data medis terbuka.
Hanya bukti.

---
⭐ 7. Pasien Pulang, Data Tetap Aman

Ketika pulang:

Data tetap 100% milik Si A.

RS hanya memegang salinan terenkripsi.

Izin akses otomatis expired setelah 60 menit.

Pasien bisa:

Lihat catatan dari RS X

Bagikan ke dokter langganan

Export untuk rumah sakit lain

Hapus izin akses dari RS X jika mau

---
🎯 III. Kenapa Ini REVOLUSIONER?

Karena tidak ada sistem kesehatan di dunia yang:

✔ pasiens-centic
✔ self-sovereign identity
✔ encrypted by default
✔ portable antar RS
✔ selective disclosure
✔ blockchain-based auditing
✔ doctor–patient direct agreement
✔ no third party

Ini menggabungkan:

kesehatan

blockchain

identitas digital

enkripsi

regulasi privasi

Dalam satu produk super jelas gunanya.
